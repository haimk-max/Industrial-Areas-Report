# תשתית שאילתות מבוססת-מסמכים (PDF Q&A Infrastructure)

> **סוג המסמך**: מסמך אדריכלות (design spec). מתאר תשתית גנרית לחלוטין למיצוי, אינדוקס ושאילתה של מסמכי דוחות סביבתיים — **לא כבולה לארכיטקטורת RAG מסוימת ולא לספק ענן מסוים**. כל בחירת מימוש (מאגר וקטורי, מודל embedding, אחסון, מנוע סינון) מסומנת `[החלטה מקומית]` ויש להשלימה בכל סביבה.
>
> **רקע**: התשתית נגזרה מאפיון פיילוט אמיתי (אזה"ת חולון–בת ים, 13 מסמכים) שנועד להתרחב לכלל הארגון. הפרמטרים והמאפיינים הספציפיים שלהלן (שדות מטה-דאטה, היררכיית מיקום, זיהוי נספחים) נלקחו מאותו אפיון, אך נוסחו כאן באופן שאינו תלוי-מימוש.

## יחס למסמכים אחרים

| מסמך | מה הוא מתאר | תחום |
|------|-------------|------|
| `docs/PDF_EXTRACTION_PROCEDURE.md` | התהליך **הקיים בפועל** בפרויקט זה | ספציפי לפרויקט |
| `docs/GENERIC_PDF_EXTRACTION_AND_RETRIEVAL.md` | מדריך גנרי לאזור חדש (שלבים 0–3 + שליפה בסיסית) | גנרי לאזור |
| **`docs/PDF_QA_INFRASTRUCTURE.md`** (מסמך זה) | **תשתית שאילתות מלאה** — אינדוקס מרוכז + שליפה לפי-דרישה + סינתזה | תשתית גנרית |

מסמך זה מרחיב את שני הקודמים: הוא לוקח את המיצוי (שלבים 0–3) והופך אותו ל**שכבת אינדקס רב-ממדי** שמיועדת לענות על **שאלות שלא ניתן לחזות מראש**, ומוסיף את שכבת השליפה-והסינתזה שמופעלת בעת שאלה.

## 1. עיקרון-העל: שני זמנים, שני טריגרים

המערכת מורכבת משני תהליכים שנפרדים בזמן ובטריגר. **ערבוב בין השניים הוא טעות התכנון הנפוצה ביותר.**

```
┌──────────────────────────────────────────────────────────────────┐
│  תהליך א׳ — עיבוד מרוכז (BATCH)                                   │
│  טריגר: מסמכים חדשים עולים למאגר המקור (object store)             │
│  תדירות: פעם אחת לכל מסמך (אידמפוטנטי)                            │
│  פלט: אינדקס עשיר, רב-ממדי, מוכן-לשאילתה                          │
│                                                                    │
│   PDFs → זיהוי-סוג → חילוץ → סינון-נספחים → נתוח → מיצוי → אינדוקס│
└──────────────────────────────────────────────────────────────────┘
                              │
                  (האינדקס נשמר, ממתין)
                              │
┌──────────────────────────────────────────────────────────────────┐
│  תהליך ב׳ — שליפה וסינתזה (ON-DEMAND)                             │
│  טריגר: שאלה מגיעה (טקסט חופשי / בחירה על מפה / סינון אזור)       │
│  תדירות: פעם לכל שאלה                                             │
│  פלט: תשובה מעמיקה, מצוטטת, בעברית                                │
│                                                                    │
│   שאלה → נתוח-כוונה → שליפה → הרכבת-קונטקסט → סינתזת-LLM → תשובה │
└──────────────────────────────────────────────────────────────────┘
```

### למה ההפרדה קריטית

- **האינדקס לא יודע מה ישאלו אותו.** לכן הוא חייב ללכוד **כל ממד שאילתה אפשרי** מראש (מיקום, זמן, מזהם, קידוח, אקוויפר, מתקן, גורם-עורך, גורם-נדון, סוג-מסמך). ככל שהאינדקס עשיר בממדים — כך יותר סוגי שאלות ייענו.
- **השאלה מגיעה אחר כך** ופוגשת אינדקס מוכן: (1) תשובה מהירה, (2) שאילתה צולבת בין מסמכים שנקלטו בזמנים שונים, (3) עומק — כי כל המסמכים כבר מעובדים.

### אנלוגיית הספרייה

ספרנית מקטלגת כל ספר שמגיע — לא לפי שאלה ספציפית, אלא לפי כל ממד שמישהו עשוי לחפש לפיו. הקורא מגיע אחר כך עם שאלה, והקטלוג מפנה אותו. **הקטלוג חייב להיות עשיר בממדים כי אינו יודע מראש מה ישאלו.**

## 2. תהליך א׳: עיבוד מרוכז ואינדוקס

הקלט: אצוות PDF ממאגר-מקור (`[החלטה מקומית]` — object store כמו S3/GCS/MinIO, או תיקייה). הפלט: אינדקס רב-ממדי.

### 2.0 קליטה ומניפסט (Intake & Catalog)

כל מסמך שנכנס נרשם ב**קטלוג מרכזי** לפני עיבוד — ה-SSOT של "מה יש לנו ומה מצבו".

```
catalog/
├── documents_manifest.jsonl      # שורה אחת לכל מסמך — מצב, hash, מטא-דאטה
└── ingestion_log.jsonl           # לוג הרצות — מתי, מה עובד, מה כשל
```

**עקרונות קליטה**:
- **קליטה אוטומטית**: העלאת PDF למאגר-המקור מפעילה קליטה (זיהוי-סוג → חילוץ → סינון → נתוח → מיצוי → אינדוקס) **ללא התערבות ידנית**.
- **הפניית-מקור נשמרת**: לכל מסמך נשמרת הפניה למיקום המקור (נתיב במאגר + שם קובץ) — תנאי לייחוס מקור בתשובות.
- **כשל-והמשך (fail-and-continue)**: קובץ פגום / מוגן-סיסמה / ללא טקסט-בר-חילוץ → נרשם ביומן עם סיבה, והקליטה **ממשיכה** לשאר הקבצים. אצווה לא נעצרת בגלל מסמך אחד.
- **סנכרון על עדכון/הסרה**: מסמך שעודכן או הוסר ממאגר-המקור → האינדקס מסונכרן כך שישקף את המצב העדכני (ראה אידמפוטנטיות ו-`superseded_by`).
- **שימור עברית RTL**: כל שרשרת העיבוד שומרת על טקסט עברי תקין (כיווניות RTL, תווים מיוחדים).

**רשומת מסמך ב-`documents_manifest.jsonl`**:

```json
{
  "doc_id": "sha256:a1b2c3...",
  "source_uri": "store://bucket/incoming/monitoring_2023_holon.pdf",
  "filename": "monitoring_2023_holon.pdf",
  "sha256": "a1b2c3...",
  "ingested_at_utc": "2026-06-14T12:00:00Z",
  "status": "indexed",
  "page_count": 23,
  "language": "he",

  "doc_type": "monitoring",
  "stages": {
    "content_type_detected": true,
    "text_extracted": true,
    "appendices_filtered": true,
    "chunked": true,
    "entities_extracted": true,
    "indexed": true
  },
  "stats": {
    "pages_extracted": 14, "pages_skipped": 9,
    "chunk_count": 31, "facts_extracted": 88,
    "ocr_pages": 0, "extraction_confidence_min": 0.97
  },
  "superseded_by": null
}
```

**אידמפוטנטיות (קריטי)**: `doc_id` = SHA-256 של תוכן הקובץ. מסמך שכבר אונדקס (אותו hash) — **מדולג**. גרסה מעודכנת (תוכן שונה) → hash חדש → רשומה חדשה, והישנה מסומנת `superseded_by`.

**סוגי מסמכים (`doc_type`)** — אנומרציה סגורה הניתנת להרחבה:
`monitoring` (ניטור) | `remediation_followup` (מעקב שיקום) | `sampling` (דיגום) | `annual` (דוח שנתי) | `historical_survey` | `permit_license` | `enforcement` | `other`

### 2.1 זיהוי סוג-תוכן וחילוץ טקסט (כולל OCR)

לפני חילוץ, המערכת מזהה **לכל עמוד** אם הוא דיגיטלי (שכבת טקסט קיימת) או סרוק/תמונה (דורש OCR). זיהוי שגוי = נתונים אבודים.

| מצב עמוד | פעולה | נימוק |
|----------|-------|-------|
| **דיגיטלי** (טקסט בר-חילוץ) | `pdftotext -layout` ישירות | חוסך עלות וזמן OCR; שומר טבלאות |
| **סרוק / איכות נמוכה** | OCR תומך-עברית (`[החלטה מקומית]` — Tesseract-heb / Vision / ABBYY) | אחרת העמוד "ריק" |

- **רמת ביטחון נשמרת**: כל חילוץ (במיוחד OCR) מתועד עם `extraction_confidence` כמטה-דאטה — לבקרת איכות (§7).
- **שמירה לפי עמוד**: הטקסט נשמר עמוד-עמוד (`pages/page_NNN.txt`) כדי שכל chunk יישא הפניית-עמוד מדויקת.
- מסמך שכל עמודיו סרוקים וללא OCR זמין → `status: "needs_ocr"`, לא נכנס לאינדקס שקט.

```
storage/<doc_id>/
├── raw.pdf
├── pages/ page_001.txt … page_023.txt
└── full.txt
```

### 2.2 סינון תוכן וזיהוי נספחים לא-רלוונטיים

לא כל עמוד שווה מיצוי. **דוחות מעבדה גולמיים** ו**יומני שדה** הם נספחים שמיצויָם מבזבז עלות ומזהם את האינדקס בטבלאות חסרות-הקשר. המערכת מזהה אותם ו**מדלגת על מיצויָם — אך שומרת רשומה על קיומם**.

#### מבנה טיפוסי של מסמך (מאפיון אמיתי)

| חלק | עמודים אופייניים | למיצוי? |
|-----|------------------|---------|
| עמוד שער (לוגו, שם, תאריך, עורכים, נמען) | 1 | חלקי (מטה-דאטה) |
| **תוכן עניינים** | 2 | כן — **אות-מפתח** לזיהוי גבולות |
| גוף: רקע, מיקום (כתובת, גוש/חלקה), תיאור טכני, מיקום בארות, ניתוח סטטיסטי (MTBE/בנזן), **טבלאות מעקב מסכמות**, סיכום והמלצות | 3–14 | **כן** |
| **נספחים — דוחות מעבדה גולמיים** (Aminolab / ALS, טבלאות צפופות, עשרות שורות, ללא פרשנות) | 15+ | **לא — דלג** |

#### אותות זיהוי (משולבים — לעולם לא אות בודד)

1. **תוכן עניינים (אות עדיף)**: אם קיים — מזהים בו את הסעיף "דוחות מעבדה"/"נספח" וקובעים את **טווח העמודים** שלו. אלו העמודים לדילוג.
2. **טבלה צפופה ללא הקשר**: עמוד עם טבלה של **>30 שורות** ושמות-עמודות כמו `Method`, `Unit`, `Result`, `Laboratory sample ID` — סבירות גבוהה לדוח-מעבדה גולמי.
3. **כותרות אופייניות (התאמה גמישה)**: "Analytical Results", "Test Report", "Certificate of Analysis" / "תוצאות מעבדה", "תעודת בדיקה", "דוח בדיקה", "נספח", "יומן שדה" — כולל וריאציות.
4. **עיתוי במסמך**: נספח החל מעמ' 15+ במסמך בן 15–25 עמ' + עמידה באותות 2–3 → סביר דוח-מעבדה.
5. **עמודים סרוקים בסוף מסמך דיגיטלי**: טפסים חוזרים / כתב-יד → מועמדים ל**יומני שדה**.

#### כללי הכרעה

- **הבחנה קריטית**: טבלאות מסכמות **בגוף** (ערכי מזהמים לאורך זמן) = רלוונטיות למיצוי; טבלאות גולמיות **בנספח** = לדילוג. אל תבלבל ביניהן.
- **מפות / גרפים / איורים**: לחלץ **רק את הכותרת/כיתוב + מספר** (למשל "מפה 1 — מיקום תחנת דלק X"), **לא** את תוכן התמונה. לשמור רשומה (סוג, מספר, עמוד) כמטה-דאטה.
- **ביטחון נמוך → בדיקה ידנית**: אם גבול-נספח אינו ודאי — **לסמן לבדיקה ידנית, לא לדלג אוטומטית**. עדיף לבזבז מיצוי מאשר לאבד מידע רלוונטי.
- **שקיפות**: לכל מסמך נשמר אילו חלקים מוצו ואילו דולגו (סוג-נספח + טווח-עמודים) — ב-`documents_manifest.jsonl`.

> **הערת אפיון**: בפיילוט שנבדק לא נמצאו יומני-שדה סרוקים (כל התוכן דיגיטלי); הנספח הלא-רלוונטי הדומיננטי היה דוחות-מעבדה. הכלל ליומני-שדה נשמר בכל זאת — אזורים/ארגונים אחרים עשויים לכלול אותם.

### 2.3 נתוח (Chunking)

הטקסט שנותר אחרי הסינון מפוצל ל-**chunks** — יחידות בר-שליפה.

| פרמטר | ערך מומלץ | נימוק |
|-------|-----------|-------|
| גודל chunk | 500–800 מילים | מספיק קונטקסט, לא בזבזני |
| חפיפה | 10–15% | מונע חיתוך משפט באמצע |
| גבול נתוח | סוף פסקה / כותרת / שורת-טבלה | לא לחתוך רעיון |
| טבלה מסכמת (גוף) | chunk שלם | טבלה חתוכה = נתונים אבודים |

כל chunk נושא `chunk_id`, `doc_id`, `page_start/end` ואת **מלוא המטה-דאטה הרב-ממדית** (§2.5).

### 2.4 מיצוי ישויות ועובדות

לכל מסמך, **סוכן AI** (persona הידרוגיאולוג) מחלץ את קטגוריות המידע המובנה (קידוחים, ממצאי-זיהום, מתקנים, הידרוגיאולוגיה, מגמות, ציטוטים, המלצות) — כמפורט ב-`GENERIC_PDF_EXTRACTION_AND_RETRIEVAL.md §2`. כל עובדה מקושרת ל-`chunk_id` + `doc_id` + עמוד, ומסווגת באמינות **A–E** (A=אומת-גולמי … E=אזכור-חלוש).

**כלל אי-המצאה**: שדה שאינו ניתן לחילוץ → מסומן `"unknown"`, **לא ממציאים ערך**.

### 2.5 שכבת המטה-דאטה והאינדוקס — לב המערכת

זהו החלק הקריטי. **האינדקס חייב ללכוד כל ממד שאילתה אפשרי מראש.** להלן סכמת המטה-דאטה המלאה — מתועדת, סגורה-ככל-הניתן, וניתנת להרחבה.

#### עקרון הנרמול הקנוני (חל על כל שדה רב-גרסתי)

מונחים נכתבים בגרסאות שונות. לכל שדה כזה נשמרים **שני ערכים**:
- `*_canonical` — הערך הקנוני המנורמל (לסינון אחיד)
- `*_original[]` — כל הגרסאות המקוריות כפי שהופיעו (לחיפוש מילולי)

דוגמאות: `PFAS` / `חומרים פלואורואלקיליים` / `PFOS,PFOA` → canonical=`PFAS`. `אזה"ת חולון` / `אזור תעשייה חולון` → canonical=`אזור תעשייה חולון`. `TCE` / `טריכלורואתילן` / `Trichloroethylene` → canonical=`Trichloroethylene`.

#### א. פרטי המסמך

| שדה | הערה |
|-----|------|
| `document_name` | שם המסמך |
| `report_date` | תאריך הדוח |
| `report_type` | ניטור / מעקב-שיקום / דיגום / שנתי / אחר (= `doc_type`) |
| `author_names[]` | שמות המחברים |
| `preparing_org` | **הגוף שהמחברים מטעמו** — חברת העריכה/הכנה |
| `subject_entity` | **הגורם שהדוח עוסק בו** (למשל "דלק הצבי", "סונול", "פז הסיירים") |

> **הבחנה קריטית**: `preparing_org` (מי כתב את הדוח — יועץ/חברת-ייעוץ) שונה מ-`subject_entity` (על מי/מה הדוח). שני שדות נפרדים. שאלה "אילו דוחות הכין יועץ X?" שונה מ"אילו דוחות עוסקים בתחנת Y?".

#### ב. גיאוהידרולוגיה

| שדה | דוגמה |
|-----|-------|
| `aquifer_names[]` | אקוויפר החוף |
| `sub_aquifers[]` | תת-אקוויפר A/B/C |
| `aquifer_units[]` | יחידות אקוויפריות |
| `monitoring_wells[]` | שמות קידוחי ניטור |
| `production_wells[]` | שמות קידוחי הפקה |

#### ג. זיהום (עם נרמול קנוני)

| שדה | דוגמה | הערה |
|-----|-------|------|
| `contaminant_groups_canonical[]` | מזהמי דלק, ממסים מוכלרים, PFAS | קבוצות מזהמים — מנורמל |
| `contaminants_canonical[]` | Benzene, Toluene, TCE, PCE | מזהם ספציפי — מנורמל |
| `*_original[]` | "בנזן", "BTEX", "פחממנים" | גרסאות מקור (כינויים, ר"ת, איות חלופי) |

#### ד. מיקום + היררכיה (ממד שליפה מרכזי)

מאפיין המיקום הוא **דרך השליפה המרכזית** (בחירה על מפה / שם-אזור). נשמרת **היררכיה**:

```
אזור / אזור-תעשייה  ←  עיר/יישוב  ←  רחוב / שם-אתר  ←  נקודת-ניטור / קידוח
```

| שדה | דוגמה |
|-----|-------|
| `area_canonical` | אזור תעשייה חולון |
| `city_canonical` | חולון |
| `street_or_site[]` | רחוב המלאכה / "תחנת דלק מרכבות האש" |
| `monitoring_points[]` | מזהי בארות/נקודות ניטור |
| `coordinates` | קווי אורך/רוחב או **רשת ישראל ITM**, אם קיימים |
| `*_original[]` | גרסאות מקור (אזה"ת חולון…) |

**כלל היררכיה**: סינון ברמה גבוהה (אזור) **כולל אוטומטית** את כל הרמות שתחתיו (עיר→רחוב→נקודה). מיקום שלא זוהה → מסומן מפורשות, **לא משויך מיקום שגוי**. קואורדינטות נחשפות כך שניתן להציגן כסמן על מפה.

#### ה. ממדים נגזרים (מהעובדות)

`content_type` (finding/trend/quote/recommendation/hydrogeology) · `evidence_class` (A–E) · `concentration`/`percent_of_standard`/`crossed_standard` · `date`/`year`/`period` · `confidence` (HIGH/MEDIUM/LOW לייחוס-מקור) · `embedding` (וקטור סמנטי).

#### מבנה האחסון — שלושה אינדקסים משלימים

```
index/
├── facts.jsonl          # (A) עובדות מובנות — סינון מטא-דאטה מדויק
├── chunks.jsonl         # (B) chunks + טקסט + מטא-דאטה — חיפוש מילולי (BM25)
└── vectors/             # (C) embeddings — חיפוש סמנטי
```

- **(A)** משרת שאילתות מדויקות ("ממצאים עם `percent_of_standard>100` ב-`area_canonical=אזור תעשייה חולון`").
- **(B)** תופס מונח מדויק ("בנזן", שם-קידוח) — כולל ה-`*_original` לתפיסת גרסאות-כתיבה.
- **(C)** תופס ניסוח חופשי ללא המילה המדויקת ("מקור צבאי" → chunk על "חומרי נפץ ממתקן תע"ש").

> **התאמה ל-RAG/ספק-ענן (`[החלטה מקומית]`)**: מימוש RAG מנוהל (למשל Knowledge Base מנוהל) ממפה ישירות: object store=מאגר-המקור; chunking+embedding=שלבים 2.3+2.5(C); metadata filtering=אינדקס (A); ההפניה למקור=`source_uri`+עמוד. סכמת המטה-דאטה כאן נשמרת בפורמט שתואם metadata-filtering של מנוע כזה. **המסמך אינו מחייב RAG** — ניתן לממש את (A)+(B) בלבד (JSONL/SQLite + BM25) ולהוסיף (C) כשצריך.

## 3. תהליך ב׳: שליפה וסינתזה (לפי שאלה)

```
שאלה → [3.1 נתוח-כוונה] → [3.2 שליפה היברידית] → [3.3 הרכבת-קונטקסט] → [3.4 סינתזה] → תשובה
                              ↑
                   [3.2ב מיקום/מפה]
```

### 3.1 נתוח כוונת השאלה

תרגום שאלה חופשית ל**מסנני-ממד** + **מונחי-חיפוש** (קריאת-LLM קצרה / heuristic), ממופה לממדי §2.5.

קלט: `"מה מצב זיהום הדלק בקידוחי תחנת מרכבות האש בחולון, והאם המגמה עולה?"`
```json
{
  "filters": {
    "area_canonical": "אזור תעשייה חולון",
    "street_or_site": "תחנת דלק מרכבות האש",
    "contaminant_groups_canonical": ["מזהמי דלק"],
    "content_type": ["contamination_finding", "trend"]
  },
  "keywords": ["בנזן", "MTBE", "BTEX", "מגמה", "עלייה"],
  "semantic_query": "מגמת זיהום דלק עולה בקידוחי תחנת מרכבות האש חולון",
  "intent": "trend_assessment",
  "requires_recent": true
}
```

### 3.2 שליפה היברידית

```
1. PRE-FILTER (מטא-דאטה)  → מצמצם לפי filters (כולל הרחבת-היררכיה: אזור⇒עיר⇒רחוב⇒נקודה)
        ▼
2. RANK (במקביל):
   ├── BM25       על chunks.jsonl   (מילות-מפתח + *_original)
   ├── Semantic   על vectors/        (דמיון ל-semantic_query)
   └── Structured על facts.jsonl    (התאמה מדויקת + מיון לפי חומרה)
        ▼
3. MERGE & RE-RANK (RRF) → הסרת כפילויות (chunk_id);
   חיזוק לפי evidence_class (A>B>C>D>E), עדכניות, חומרה
        ▼
   top-K chunks (~10-20) + top-M facts (~30-50)
```
**עיקרון anti-tunnel-vision**: תמיד לשלוף משלושת המקורות. שאלה "TCE" (מילה) עם תשובה ב-chunk "טריכלורואתילן" (סמנטי) — רק השילוב תופס.

### 3.2ב שליפה לפי מיקום ומפה

ערוץ שליפה מרכזי בפני עצמו:
- **בחירה על מפה / שם-אזור** → החזרת **כל** המסמכים והקטעים המשויכים למיקום ו**לכל הרמות שתחתיו** בהיררכיה.
- **מאפיין-אזור טקסטואלי** ("חולון", "רחוב המלאכה, חולון") → התאמה לערך הקנוני (`*_canonical` דרך `*_original`) → תוצאות תואמות.
- **קואורדינטות** נחשפות כסמני-מפה.
- **שילוב מיקום + שאלה** → סינון-מיקום ⨯ שליפה-סמנטית, תשובה ממוקדת-מיקום.
- **אין מסמכים לאזור** → הודעה מפורשת "אין מידע זמין עבור מיקום זה" (לא תשובה ריקה/מומצאת).

### 3.3 הרכבת קונטקסט

ה-chunks והעובדות מורכבים לפורמט LLM תחת **תקציב-טוקנים**:

```markdown
## עובדות מובנות רלוונטיות
| מזהם | קידוח | ריכוז | % מתקן | תאריך | מקור | עמ' | אמינות |
|------|-------|-------|--------|-------|------|-----|--------|
| Benzene | נת חולון 7 | 320 µg/L | 3200% | 2023-04 | monitoring_2023.pdf | 9 | B |

## קטעי טקסט מקוריים
### [מקור 1] monitoring_2023.pdf, עמ' 9 (אמינות B) — אזור תעשייה חולון / מרכבות האש
"בקידוח נת חולון 7 נמדד בנזן בריכוז 320 מק\"ג/ל..."
```
**ניהול תקציב** (`[החלטה מקומית]`): ~70% קטעי-טקסט / ~20% עובדות / ~10% רזרבה. חריגה → קיצוץ לפי דירוג (לא חיתוך chunk באמצע). **כל קטע נושא אטריבוציה מלאה** (קובץ+עמוד+מיקום+אמינות).

### 3.4 סינתזה (LLM)

`[החלטה מקומית]` — Opus (עומק) / Sonnet (מהירות). עקרונות ה-system prompt:
- **persona**: הידרוגיאולוג מומחה; **תשובה בעברית**.
- **ציטוט חובה**: כל קביעה → מקור (`[monitoring_2023.pdf, עמ' 9]`). אין מקור → לא נכתב.
- **כיבוד אמינות**: D/E כ"השערה"; A/B ניתן לקבוע. ייחוס-מקור עם HIGH/MEDIUM/LOW.
- **אי-המצאה**: אין במאגר → "המידע אינו זמין במסמכים שנשלפו". לא להשלים מהזיכרון.
- **שיחה רב-תורית**: שימור הקשר בין שאלות עוקבות (הכוונה הקודמת + הישויות שאוזכרו נשמרים לתור הבא).

```markdown
## תשובה
<סינתזה מעמיקה, מצוטטת>
## מקורות
- [1] monitoring_2023.pdf, עמ' 9 — אזור תעשייה חולון
## רמת ודאות והסתייגויות
<פערים, הנחות, אמינות נמוכה>
```

     │ 2.3 נתוח            → chunks (טבלה-מסכמת שלמה)

     ▼

═════════ תהליך ב׳: שליפה (ON-DEMAND) ═════════
  שאלה / בחירת-מפה / סינון-אזור
     │ 3.1 נתוח-כוונה   → filters + keywords + semantic_query

     │ 3.3 הרכבת-קונטקסט → facts table + excerpts + אטריבוציה (תקציב-טוקנים)
     │ 3.4 סינתזת-LLM   → persona + ציטוט + אמינות + אי-המצאה + רב-תורי
     ▼
  תשובה מעמיקה, מצוטטת, בעברית
```

## 5. עקרונות תכנון (Design Invariants)

1. **אינדוקס רחב מראש** — כל ממד פוטנציאלי נכנס בקליטה. הוספת ממד = עיבוד-מחדש. ספק עדיף.
2. **אטריבוציה מקצה-לקצה** — chunk→doc+עמוד; עובדה→chunk_id. בלעדיה אין ציטוט.
3. **אידמפוטנטיות לפי hash** — קליטה חוזרת לא משכפלת; גרסה חדשה=hash חדש+`superseded_by`; סנכרון על עדכון/הסרה.
4. **הפרדת batch/on-demand** — עיבוד לעולם לא בזמן-שאלה.
5. **שליפה היברידית תמיד** — מטא-דאטה + מילולי + סמנטי.
6. **נרמול קנוני + שימור-מקור** — כל שדה רב-גרסתי: `*_canonical` לסינון, `*_original[]` לחיפוש.
7. **הבחנת עורך/נדון** — `preparing_org` ≠ `subject_entity`.
8. **מיקום היררכי** — סינון-אזור כולל את כל הרמות שתחתיו; מיקום-לא-מזוהה מסומן, לא מומצא.
9. **כשל-והמשך** — מסמך בעייתי נרשם ומדולג; האצווה לא נעצרת.
10. **אמינות זורמת לתשובה** — A–E משפיע על דירוג ועל ניסוח (עובדה↔השערה).
11. **אי-המצאה** — מחוץ-לקונטקסט = "לא זמין".
12. **דלג-אך-רשום** — נספח שדולג עדיין מתועד (סוג+טווח-עמודים); ביטחון-נמוך→בדיקה-ידנית.

## 6. מדרגיות, הרשאות ונראות

### מדרגיות (פיילוט → ארגון)
- תוספת מסמכים = אותו תהליך קליטה ממאגר-המקור, **ללא שינוי ארכיטקטורה**.
- סכמת המטה-דאטה והמיקום **ניתנת להרחבה** לסוגי-מסמכים ואזורים נוספים (אנומרציות פתוחות-להרחבה, לא קשיחות).
- `[החלטה מקומית]` — יעדי זמן-שליפה/עלות נקבעים בעיצוב; מעבר מ-JSONL-בזיכרון ל-DB/וקטור-מנוהל בצמיחה.

### הרשאות
- `[החלטה מקומית]` — הפרדת הרשאות-גישה למסמכים לפי משתמש/קבוצה (תשתית לשלב הארגוני). מומלץ לשמור `access_group` כשדה מטה-דאטה ולסנן בו ב-pre-filter.

### נראות ובקרת איכות
- **סיכום קליטה**: בסיום אצווה — כמה נקלטו בהצלחה / כמה נכשלו (+סיבות) מ-`ingestion_log.jsonl`.
- **סקירת חילוץ**: ניתן לסקור את המטה-דאטה והמיקום שחולצו לכל מסמך (אימות ידני בפיילוט).
- **דגלי איכות**: `extraction_confidence` מתחת לסף, או גבול-נספח לא-ודאי → המסמך מסומן לבדיקה ידנית (לא מדולג שקט).

## 7. נקודות-החלטה מקומיות (`[החלטה מקומית]`)

| # | החלטה | אפשרויות | ברירת-מחדל |
|---|-------|----------|------------|
| 1 | מאגר-מקור | S3 / GCS / MinIO / תיקייה | object store קיים |
| 2 | OCR עברית | Tesseract-heb / Vision / ABBYY | Tesseract-heb |
| 3 | אחסון עובדות | JSONL / SQLite / DuckDB / Postgres | JSONL → SQLite בצמיחה |
| 4 | מודל embedding | Voyage / OpenAI / מקומי / מנוהל | מקומי-קל בהתחלה |
| 5 | מאגר וקטורי | FAISS / Chroma / pgvector / Qdrant / RAG-מנוהל | cosine-בזיכרון → FAISS |
| 6 | מנוע BM25 | rank-bm25 / Tantivy / FTS / Elasticsearch | rank-bm25 |
| 7 | מיזוג דירוג | RRF / משוקלל-ידני | RRF |
| 8 | מודל סינתזה | Opus / Sonnet | Opus לדוחות, Sonnet לאד-הוק |
| 9 | הרשאות | per-user / per-group / ACL | `access_group` במטה-דאטה |

**מסלול-התחלה מינימלי**: object-store קיים → `pdftotext`+OCR → סינון-נספחים → JSONL (facts+chunks) בזיכרון → BM25 (`rank-bm25`) **בלי** שכבה וקטורית → סינתזה ב-Opus. מוסיפים וקטור (4+5) כשחיפוש-מילולי מתחיל לפספס ניסוחים-חופשיים. **כל זאת ללא RAG מנוהל** — שכבת RAG/ספק-ענן היא שדרוג אופציונלי, לא תנאי.

## 8. Checklist הפעלה

**עיבוד (batch)**:
- [ ] קטלוג מאותחל; `doc_id`(hash) + `source_uri` + `doc_type` לכל מסמך; אידמפוטנטיות מאומתת
- [ ] זיהוי דיגיטלי/סרוק עובד; OCR-עברית למסמכים סרוקים; `extraction_confidence` נשמר
- [ ] סינון-נספחים: TOC + טבלאות-צפופות(>30) + כותרות + עיתוי; דלג-אך-רשום; ביטחון-נמוך→ידני
- [ ] מפות/גרפים → כותרת-בלבד; טבלאות-מסכמות(גוף) נשמרות, גולמיות(נספח) מדולגות
- [ ] נתוח שומר טבלאות-מסכמות שלמות; כל chunk נושא עמוד
- [ ] מיצוי: עובדה→chunk_id; A–E; שדות לא-נמצאו = `"unknown"` (לא המצאה)
- [ ] מטה-דאטה: `preparing_org`≠`subject_entity`; `*_canonical`+`*_original`; היררכיית-מיקום; קואורדינטות
- [ ] שלושת האינדקסים נכתבו (או לפחות facts+chunks)
- [ ] כשל-והמשך מאומת; סנכרון על עדכון/הסרה

**שליפה (on-demand)**:
- [ ] נתוח-כוונה ממפה שאלת-בדיקה למסננים נכונים (כולל מיקום)
- [ ] סינון-אזור מרחיב לכל הרמות שתחתיו בהיררכיה
- [ ] בחירת-מפה / שם-אזור מחזירה מסמכים+קטעים; קואורדינטות חשופות לסמן
- [ ] שליפה מחזירה משלושת המקורות
- [ ] קונטקסט נושא אטריבוציה מלאה לכל קטע
- [ ] תשובה בעברית, מצטטת מקור לכל קביעה; שיחה רב-תורית שומרת הקשר
- [ ] שאלה ללא תשובה / אזור ריק → "לא זמין" (לא המצאה)
```
