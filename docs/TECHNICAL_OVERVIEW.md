# מסמך טכני — ארכיטקטורת המערכת לניתוח זיהום מי תהום
### לאנשי פיתוח ומערכות מידע

> מסמך זה מיועד לאנשי פיתוח ומערכות מידע של השירות ההידרולוגי.
> **מטרה**: לתאר את הארכיטקטורה, מקורות המידע, חלוקת העבודה בין רכיבים דטרמיניסטיים ל-LLM, ושכבת הניהול והבקרה האוטומטית.
> **מצב**: תשתית הושלמה; אומתה על שני אזורים (רעננה, חולון). הפייפליין גנרי (`--zone`).

---

## 1. תמונת-על אדריכלית

המערכת היא **pipeline היברידי** המורכב משלוש שכבות לוגיות:

```
┌─────────────────────────────────────────────────────────────────────┐
│  שכבת קלט              שכבת עיבוד               שכבת בקרה ופלט         │
│  ─────────             ──────────               ──────────────        │
│                                                                       │
│  Excel/CSV   ──┐                                                      │
│  PDF         ──┼─▶  [דטרמיניסטי]  ──▶  [LLM מבוקר]  ──▶  [QA Gates]   │
│  מידע מקורות ──┘     parse/trend       Opus #1/#2/#3     8 שערים      │
│                      forensics         (prompt בנוי)    ──▶ תוצרים    │
│                                                                       │
└─────────────────────────────────────────────────────────────────────┘
```

**עקרון-העל**: כל שלב הוא או **דטרמיניסטי לחלוטין** (אותו קלט  — אותו פלט, ניתן לבדיקה ב-CI) או **גבול-LLM מפורש** (Opus call). אין "אזורים אפורים". כל גבול-LLM ניזון מ-prompt בנוי שנגזר אוטומטית מהשלב הקודם, ומאומת בשער QA לפני ואחרי.

**Driver**: `scripts/run_pipeline.sh <ZONE> <stage>` מריץ את השלבים הדטרמיניסטיים ועוצר בכל גבול-LLM עם הוראת-המשך. ה-SSOT לתהליך: `ZONE_REPORT_PROCESS_GUIDE.md` §VIII; מפת ההרצה: `ORCHESTRATION.md`.

## 2. מקורות המידע

| מקור | פורמט | תפקיד | קלט עתידי |
|------|-------|--------|-----------|
| מדידות ניטור | Excel / CSV | סדרת-זמן של ריכוזי מזהמים לכל קידוח × פרמטר | **חיבור ישיר ל-DB** של מערכת המידע |
| דו"חות קודמים | PDF | הקשר היסטורי, מסקנות קודמות, הידרוגאולוגיה | מאגר מסמכים מתויג / RAG |
| מידע על מקורות זיהום | PDF / web | זיהוי מתקנים חשודים, חתימות מזהמים | מאגר מקורות + חיפוש מובנה |

**הערה ארכיטקטונית**: שכבת הקלט מבודדת מאחורי `parse_excel.py` + `extract_zone_pdfs.py`. החלפת מקור הקלט (קבצים  — חיבור ישיר) משפיעה **רק על שכבה זו**; שאר הפייפליין צורך את ה-Data Pack המתוקנן (סעיף 3) ואינו מודע למקור.

**אידמפוטנטיות חילוץ PDF**: כל קובץ נרשם ב-`_pdf_index.json` עם `extraction_ok` + timestamp. הרצה חוזרת מדלגת על קבצים שכבר חולצו אלא אם `--force`.

## 3. השלב הדטרמיניסטי — ה-Structured Data Pack

הליבה הדטרמיניסטית מייצרת **6–7 קבצי CSV מתוקננים** ב-`{Zone}/02_data/`. זהו **החוזה** בין העיבוד הדטרמיניסטי לבין כל מה שבא אחריו (LLM + מחוללי דו"ח). הסכמה המלאה: `DATA_PIPELINE_SPEC.md`.

| CSV | תוכן | מפתח |
|-----|------|-------|
| `measurements_scoped.csv` | כל מדידה בקידוחי האזור + מקור (source_file, page/row) | well × parameter × date |
| `latest_results.csv` | תוצאה אחרונה לכל קידוח × פרמטר + יחס-לתקן + severity | well × parameter |
| `severity_by_well_family.csv` | אינדקס חומרה לפי **C_max_5y** (חלון 5 שנים), לא תוצאה אחרונה | well × family |
| `trends_by_well_parameter.csv` | תוצאות Mann-Kendall (Z, p, SNR, soft_trigger) | well × parameter |
| `monitoring_gaps.csv` | פערי כיסוי מרחביים/פרמטריים | well / parameter |
| `figure_ready_series.csv` | סדרות מוכנות-לציור (ללא חישוב במחולל) | well × parameter × date |

**מתודולוגיה סטטיסטית (קבועה, לא ניתנת לשינוי ללא אישור מומחה):**

- **מגמות**: Mann-Kendall (tie-corrected variance, continuity-corrected Z) עם SNR gating (≥0.3), soft_trigger = 2 ערכים עולים רצופים בחלון 5 שנים. **לא** רגרסיה לינארית, לא ML.
- **חומרה**: `bucket(C_max_5y / DWS × 100)`, סולם 0–8.
- **פורנזיקה**: שרשראות פירוק, חתימות מקור, זוגות co-occurrence — הסתברותי, עם רמת ביטחון (HIGH/MEDIUM/LOW). אינו ייחוס דטרמיניסטי.
- **הדרה**: TPFAS ו-BETK תמיד מודרים (סכומים מחושבים). species בודדים (PFHxS, PFOA) הם הייצוג הקנוני.

**למה דטרמיניסטי דווקא כאן**: החישובים האלה חייבים להיות מדויקים, ניתנים-לאימות מול דו"ח-המקור, וחזרתיים byte-identical. הם **אינם** מתאימים ל-LLM — לא רוצים ש"כמעט אותו מספר" יחזור בכל הרצה. הם העוגן שעליו נשען השיפוט.

**סקריפטים** (כולם `--zone`):
```bash
parse_excel.<bdi>py → trend_analysis</bdi>.py → forensics_analyzer.py → generate_zone_data_pack.py
```

## 4. השלב ה-LLM — שלושה גבולות Opus מבוקרים

המשימות הדורשות שיפוט והקשר — שאינן נגזרות דטרמיניסטית מ-CSV — מבוצעות ע"י Opus בשלושה גבולות מוגדרים:

| Opus call | קלט | פלט | למה LLM |
|-----------|-----|------|----------|
| **#1 — אבחון אזורי** | Data Pack + Context Pack | `04_diagnosis/zone_diagnosis.md` (8 שאלות + טבלת `סדר מוקדים`) | אשכול גיאוגרפי של מוקדי זיהום + קישור מנגנוני אינם דטרמיניסטיים מ-CSV |
| **#2 — דו"ח מומחה** | Data Pack + diagnosis + precedent | `output/{ZONE}_REPORT_Vn.md` (6 פרקים + נספחים) | חיבור בין ממצאים, ניסוח מקצועי, נרטיב |
| **#3 — brief ניהולי** | הדו"ח המאושר | `raw_brief.yaml`  — דו"חות ניהוליים | המרת דו"ח עשיר לתקציר דו-קהלי |

### העיקרון הקריטי: prompt בנוי, לא שאלה פתוחה

ה-LLM **אינו** מקבל הנחיה חופשית. כל prompt הוא **ארטיפקט נגזר (derived)** המיוצר ע"י `render_zone_prompt.py`:

- נגזר מתבנית קנונית (`scripts/templates/zone_report_prompt_template_v5.md`)
- מוזרק עם metadata מה-Data Pack (ספירת קידוחים, מדידות, טווח שנים)
- מוזרק עם `{FOCUS_ORDER_LIST}` שנלקח מטבלת האבחון (חוזה בין-שלבי)
- מוזרק עם בלוק הטרמינולוגיה מ-SSOT יחיד (`docs/STYLE_GUIDE.md §B.5`)
- נושא **חותמת provenance**: `template_sha256_12` + `diagnosis_sha256_12`

> **כלל-זהב**: prompt מרונדר הוא *materialized copy*, לא reference. הוא נוטה לסחוף (drift) בשקט. לכן הוא (א) מתחדש מה-SSOT על-פי דרישה, ו-(ב) נבדק לטריות ב-Gate 3 (השוואת sha חי מול חתום). זה מה שמונע ש-Opus יצרוך snapshot מיושן.

## 5. שכבת הניהול והבקרה — 8 שערי QA

זוהי השכבה שהופכת תוצר-LLM ל**תוצר אמין בסביבת ייצור**. `scripts/qa_pipeline.py` מריץ שערים אוטומטיים שכל אחד **חוסם** המשך אם החוזה מופר.

| שער | שלב נבדק | מה נאכף | חסימה |
|-----|-----------|---------|--------|
| **Gate 2** | Data Pack | 6 CSVs קיימים, סכמה תקינה, כל נתון עם מקור, אין פערים שקטים | FAIL |
| **Gate 3** | שכבת prompt | `template_sha` + `diagnosis_sha` תואמים את ה-SSOT החי (טריות) | FAIL אם מיושן |
| **Gate 4** | אבחון | בלוק `## סדר מוקדים` קיים, פרסבילי, ממוין לפי חומרת-מוקד | FAIL |
| **Gate 5** | דו"ח | מבנה 6 פרקים, ספירת קידוחים עקבית, טרמינולוגיה, focus-first, PFAS ב"פערי כיסוי" | FAIL |
| **Gate 6** | HTML/Word | מספר מעגלי-קידוח במפה ≥80% מהקידוחים בscope, bounds דינמיים | FAIL |
| **Gate 8** | דו"חות ניהוליים | brief↔report sha תואם (טריות); אנונימיזציה (שמות מתקנים ב-PUBLIC) | FAIL |

(Gate 7 = `validate` = הרצת all.)

### שני חוזי-staleness קריטיים (footguns שתועדו ונאכפו)

1. **<bdi>diagnosis → prompt</bdi>** (Gate 3): הרצה חוזרת של שלב 4 (אבחון) ללא רינדור-מחדש של prompt הדו"ח  — Opus צורך snapshot מיושן של `focus_order`. **נאכף**: ה-prompt חותם `diagnosis_sha256_12`; Gate 3 משווה לקובץ האבחון החי.

2. **<bdi>report → brief</bdi>** (Gate 8): שינוי בדו"ח-המקור ללא רענון ה-brief  — דו"חות ניהוליים מיושנים. **נאכף**: ה-brief נושא `source_report_sha256_12`; Gate 8 משווה לדו"ח האחרון.

### אכיפת אנונימיזציה (Gate 8)

הגרסה הציבורית נסרקת מול blocklist (`sources[].name_internal`):
- **שם מתקן** ב-PUBLIC  — **FAIL** (חוסם פרסום)
- **שם קידוח אמיתי** ב-PUBLIC  — **WARN** (מומלץ קוד B-NN; חריגים מאושרים אפשריים)

זה מונע הישנות של דליפת מידע — לא הסתמכות על זיכרון אנושי.

## 6. למה השילוב הזה — ולא אחד מהקצוות

| גישה | בעיה |
|------|------|
| **הכל דטרמיניסטי** | לא ניתן לבצע אשכול גיאוגרפי, חיבור ממצאים, ניסוח מקצועי — דורש שיפוט |
| **הכל LLM** | לא חזרתי, לא ניתן לאימות, "כמעט נכון" בחישובים קריטיים, סיכון לדליפה |
| **היברידי + שערים** | דיוק במקום שצריך + שיפוט במקום שצריך + חוזה אכוף ביניהם |

**המסקנה ההנדסית בת-ההכללה**: תוצר GenAI אמין בסביבת ייצור = *דטרמיניזם בליבה* + *LLM בגבולות מוגדרים* + *שערי בקרה שאוכפים את החוזה*. הדפוס אינו ייחודי לזיהום מי תהום — הוא ישים לכל זרימת-עבודה עם מסה של מידע מקצועי וצורך בתוצר עקבי ומבוקר.

## 7. גנריות, בדיקות, ותחזוקתיות

**גנריות (`--zone`)**: הפייפליין אינו נעוץ-בחולון. כל הסקריפטים מקבלים `--zone` וגוזרים נתיבים מהשם. אומת על רעננה (7 קידוחים) וחולון (111 קידוחים) — סף Gate 6 הוא zone-relative (≥80% מהקידוחים), לא קבוע.

**מוסכמת casing** (חוב מתועד, REQ #28): סקריפטי-ליבה מקבלים `--zone holon` (אות קטנה); כלי-V5 מקבלים `--zone Holon` (שם-ספרייה). מתועד ב-`ORCHESTRATION.md`.

**בדיקות**: `tests/` כולל בדיקות יחידה + 3 validators לדו"ח (chart_refs, tone, attribution) + 13 בדיקות למחולל ה-brief. כל שינוי במחולל מאומת מול reference frozen (רגנרציה byte-identical).

**Audit trail**: כל נתון  — מקור (source_file + page/row); כל prompt  — חותמת sha; כל גרסת דו"ח  — V-number; כל ארטיפקט נגזר מסומן `DO NOT EDIT BY HAND`.

**מצב גנריות הסקריפטים** (חוב 18-האזורים) מתועד בטבלה ב-`ORCHESTRATION.md`. סקריפטי legacy (V4-era) ב-`scripts/legacy/` — ARCHIVED.

## 8. רצף הרצה מלא (חולון, לדוגמה)

```bash
ZONE_NAME_HE='אזה״ת חולון' scripts/run_pipeline.sh Holon data
#   <bdi>parse_excel → trend</bdi> → forensics → data_pack → Gate 2
# … הרכבת 03_context/ ידנית (NotebookLM-like) …

ZONE_NAME_HE='אזה״ת חולון' scripts/run_pipeline.sh Holon render-diagnosis  # Gate 3
# … OPUS #1 → zone_diagnosis.md … ; qa_pipeline --gate 4

ZONE_NAME_HE='אזה״ת חולון' scripts/run_pipeline.sh Holon render-report     # Gate 3 (טריות)
# … OPUS #2 → HOLON_REPORT_Vn.md … ; qa_pipeline --gate 5

scripts/run_pipeline.sh Holon html        # generate_holon_v5_html.py (auto-detect Vn) ; Gate 6
scripts/run_pipeline.sh Holon validate     # Gate all

# … רק אחרי אישור מומחה: …
scripts/run_pipeline.sh Holon exec-summary # 8a <bdi>prepare → OPUS</bdi> #3 → finalize → html-from-brief ; Gate 8
```

**גבולות Opus** (לא ניתן לאטמט): render-diagnosis —#1, render-report —#2, exec-summary —#3. הדרייבר עוצר שם.

## 9. נקודות פתוחות (chunks לפיתוח עתידי)

| נושא | מצב | רכיב |
|------|------|-------|
| חיבור ישיר ל-DB | מתוכנן | שכבת הקלט (סעיף 2) |
| RAG על מאגר דו"חות | נדחה (מתועד) | Context Pack assembly |
| OCR ל-PDF סרוקים | חוסם נתונים (TAHAL 2008) | `extract_zone_pdfs.py` |
| איחוד מוסכמת `--zone` | חוב REQ #28 | `cli_common.py` + ליבה |
| basemap לאריחי-מפה | נדחה (סביבה) | `svg_charts.py` |

**הרכיבים הניתנים לשימוש-חוזר** (מחוץ לתחום זיהום מי תהום):
- מנוע ניתוח דטרמיניסטי (מגמות/חומרה)  — כל סדרת-זמן סביבתית
- דפוס LLM מבוקר (prompt בנוי + שערים)  — כל משימת ניסוח/סיווג עקבית
- מחולל דו-קהלי (פנימי+ציבורי מאותו מקור)  — כל תוצר עם גרסת-פרסום
- שכבת QA (audit trail, אנונימיזציה, staleness)  — כל זרימה רגולטורית

---

> **לסיכום הנדסי**: הערך אינו במחולל-דו"ח חד-פעמי, אלא ב**ארכיטקטורה** — הפרדה נקייה בין דטרמיניזם ל-LLM, חוזים אכופים בין שלבים, ושכבת בקרה שמונעת drift, staleness ודליפה. זה מה שהופך את הכלי ממדגם-הוכחה (POC) לרכיב תשתית שניתן לתחזק, להרחיב ל-18 אזורים, ולהטמיע במערכות הרשות.
