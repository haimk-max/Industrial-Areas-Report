לאנשי פיתוח ומערכות מידע
מטרה: לתאר את הארכיטקטורה, מקורות המידע, חלוקת העבודה בין רכיבים דטרמיניסטיים ל-LLM, ושכבת הניהול והבקרה האוטומטית.

תמונת-על אדריכלית

המערכת היא pipeline היברידי המורכב משלוש שכבות לוגיות. הזרימה נקראת מימין לשמאל: מקורות הקלט (מימין) זורמים דרך עיבוד דטרמיניסטי, שלושה גבולות LLM מבוקרים, ושכבת בקרה — אל התוצרים (משמאל).

עקרון-העל: כל שלב הוא או דטרמיניסטי לחלוטין (אותו קלט ← אותו פלט, ניתן לבדיקה ב-CI) או גבול-LLM מפורש (Opus call). אין "אזורים אפורים". כל גבול-LLM ניזון מ-prompt בנוי שנגזר אוטומטית מהשלב הקודם, ומאומת בשער QA לפני ואחרי.

3

השלב הדטרמיניסטי — ה-Structured Data Pack

הליבה הדטרמיניסטית מייצרת 6–7 קבצי CSV מתוקננים ב-{Zone}/02_data/. זהו החוזה בין העיבוד הדטרמיניסטי לבין כל מה שבא אחריו (LLM + מחוללי דו"ח). הסכמה המלאה: DATA_PIPELINE_SPEC.md.

CSVתוכןמפתח
measurements_scopedכל מדידה בקידוחי האזור + מקור (source_file, page/row)well × parameter × date
latest_resultsתוצאה אחרונה לכל קידוח × פרמטר + יחס-לתקן + severitywell × parameter
severity_by_well_familyאינדקס חומרה לפי C_max_5y (חלון 5 שנים), לא תוצאה אחרונהwell × family
trends_by_well_parameterתוצאות Mann-Kendall (Z, p, SNR, soft_trigger)well × parameter
monitoring_gapsפערי כיסוי מרחביים/פרמטרייםwell / parameter
figure_ready_seriesסדרות מוכנות-לציור (ללא חישוב במחולל)well × parameter × date

מתודולוגיה סטטיסטית (קבועה, לא ניתנת לשינוי ללא אישור מומחה)

  • מגמות: Mann-Kendall (tie-corrected variance, continuity-corrected Z) עם SNR gating (≥0.3), soft_trigger = 2 ערכים עולים רצופים בחלון 5 שנים. לא רגרסיה לינארית, לא ML.
  • חומרה: bucket(C_max_5y / DWS × 100), סולם 0–8.
  • פורנזיקה: שרשראות פירוק, חתימות מקור, זוגות co-occurrence — הסתברותי, עם רמת ביטחון (HIGH/MEDIUM/LOW). אינו ייחוס דטרמיניסטי.
  • הדרה: TPFAS ו-BETK תמיד מודרים (סכומים מחושבים). species בודדים (PFHxS, PFOA) הם הייצוג הקנוני.

למה דטרמיניסטי דווקא כאן: החישובים האלה חייבים להיות מדויקים, ניתנים-לאימות מול דו"ח-המקור, וחזרתיים byte-identical. הם אינם מתאימים ל-LLM — לא רוצים ש"כמעט אותו מספר" יחזור בכל הרצה. הם העוגן שעליו נשען השיפוט.

parse_excel.py → trend_analysis.py → forensics_analyzer.py → generate_zone_data_pack.py

השלב ה-LLM — שלושה גבולות Opus מבוקרים

המשימות הדורשות שיפוט והקשר — שאינן נגזרות דטרמיניסטית מ-CSV — מבוצעות ע"י Opus בשלושה גבולות מוגדרים:

Opus callקלטפלטלמה LLM
#1 — אבחון אזוריData Pack + Context Packzone_diagnosis.md (8 שאלות + טבלת סדר מוקדים)אשכול גיאוגרפי + קישור מנגנוני אינם דטרמיניסטיים מ-CSV
#2 — דו"ח מומחהData Pack + diagnosis + precedent{ZONE}_REPORT_Vn.md (6 פרקים + נספחים)חיבור בין ממצאים, ניסוח מקצועי, נרטיב
#3 — brief ניהוליהדו"ח המאושרraw_brief.yaml → דו"חות ניהולייםהמרת דו"ח עשיר לתקציר דו-קהלי

העיקרון הקריטי: prompt בנוי, לא שאלה פתוחה

ה-LLM אינו מקבל הנחיה חופשית. כל prompt הוא ארטיפקט נגזר (derived) המיוצר ע"י render_zone_prompt.py:

  • נגזר מתבנית קנונית (zone_report_prompt_template_v5.md)
  • מוזרק עם metadata מה-Data Pack (ספירת קידוחים, מדידות, טווח שנים)
  • מוזרק עם {FOCUS_ORDER_LIST} שנלקח מטבלת האבחון (חוזה בין-שלבי)
  • מוזרק עם בלוק הטרמינולוגיה מ-SSOT יחיד (docs/STYLE_GUIDE.md §B.5)
  • נושא חותמת provenance: template_sha256_12 + diagnosis_sha256_12
כלל-זהב: prompt מרונדר הוא materialized copy, לא reference. הוא נוטה לסחוף (drift) בשקט. לכן הוא (א) מתחדש מה-SSOT על-פי דרישה, ו-(ב) נבדק לטריות ב-Gate 3 (השוואת sha חי מול חתום). זה מה שמונע ש-Opus יצרוך snapshot מיושן.

מבנה הפרומפט — נקודות ההזרקה

במקום להטמיע כאן את טקסט הפרומפט המלא (שהוא נגזר ומשתנה בין אזורים), להלן מה מוזרק לתבנית הקנונית בעת הרינדור. כל אסימון {...} בתבנית מוחלף בערך שנשאב מ-SSOT חי — לא מטקסט מועתק:

אסימון בתבניתמקור (SSOT)תוכן מוזרק
{TOTAL_ACTIVE} {N_MEASUREMENTS} {YEAR_START/END}02_data/measurements_scoped.csvספירת קידוחים מנותחים, מדידות, טווח שנים — נשאב מה-Data Pack (לא ידני)
{FOCUS_ORDER_LIST}טבלת סדר מוקדים ב-zone_diagnosis.mdסדר המוקדים הגיאוגרפיים (החוזה הבין-שלבי אבחון←דו"ח)
{TERMINOLOGY_BLOCK}docs/STYLE_GUIDE.md §B.5בלוק ההחלפות הקנוני — אותו SSOT ש-Gate 5 אוכף
{ZONE} {ZONE_NAME_HE} {PRECEDENT_ZONE}CLI argsשם האזור, שם עברי, אזור-תקדים סגנוני

התבנית הקנונית עצמה (מבנה 6 פרקים + <focus_order> + <figure_rules>) שמורה ב-scripts/templates/zone_report_prompt_template_v5.md; הרינדור ב-scripts/render_zone_prompt.py. הפרומפט המרונדר נכתב ל-{Zone}/context_pack/05_prompt/ ונושא בראשו חותמת provenance:

<!-- RENDERED ARTIFACT — DO NOT EDIT BY HAND.
     Generated by scripts/render_zone_prompt.py --step report …
     template=zone_report_prompt_template_v5.md
     template_sha256_12=a1b2c3d4e5f6
     diagnosis_sha256_12=9f8e7d6c5b4a
     total_active=111  measurements=15173  years=2008-2025
     Re-run after ANY change to the template or the zone diagnosis. -->

שתי החותמות (template_sha, diagnosis_sha) הן בדיוק מה ש-Gate 3 משווה מול ה-SSOT החי — כך snapshot מיושן נחסם אוטומטית.

שכבת הניהול והבקרה — 8 שערי QA

זוהי השכבה שהופכת תוצר-LLM לתוצר אמין בסביבת ייצור. scripts/qa_pipeline.py מריץ שערים אוטומטיים שכל אחד חוסם המשך אם החוזה מופר.

שערשלב נבדקמה נאכףחסימה
Gate 2Data Pack6 CSVs קיימים, סכמה תקינה, כל נתון עם מקור, אין פערים שקטיםFAIL
Gate 3שכבת prompttemplate_sha + diagnosis_sha תואמים את ה-SSOT החי (טריות)FAIL

שני חוזי-staleness קריטיים (footguns שתועדו ונאכפו)

  1. diagnosis → prompt (Gate 3): הרצה חוזרת של שלב 4 (אבחון) ללא רינדור-מחדש של prompt הדו"ח → Opus צורך snapshot מיושן של focus_order. נאכף: ה-prompt חותם diagnosis_sha256_12; Gate 3 משווה לקובץ האבחון החי.
  2. report → brief (Gate 8): שינוי בדו"ח-המקור ללא רענון ה-brief → דו"חות ניהוליים מיושנים. נאכף: ה-brief נושא source_report_sha256_12; Gate 8 משווה לדו"ח האחרון.

אכיפת אנונימיזציה (Gate 8)

הגרסה הציבורית נסרקת מול blocklist (sources[].name_internal):

  • שם מתקן ב-PUBLIC → FAIL (חוסם פרסום)
  • שם קידוח אמיתי ב-PUBLIC → WARN (מומלץ קוד B-NN; חריגים מאושרים אפשריים)

זה מונע הישנות של דליפת מידע — לא הסתמכות על זיכרון אנושי.

7

גנריות, בדיקות, ותחזוקתיות

  • גנריות (--zone): הפייפליין אינו נעוץ-בחולון. כל הסקריפטים מקבלים --zone וגוזרים נתיבים מהשם. אומת על רעננה (7 קידוחים) וחולון (111 קידוחים) — סף Gate 6 הוא zone-relative (≥80% מהקידוחים), לא קבוע.
  • מוסכמת casing (חוב מתועד, REQ #28): סקריפטי-ליבה מקבלים --zone holon (אות קטנה); כלי-V5 מקבלים --zone Holon (שם-ספרייה). מתועד ב-ORCHESTRATION.md.
  • בדיקות: tests/ כולל בדיקות יחידה + 3 validators לדו"ח (chart_refs, tone, attribution) + 13 בדיקות למחולל ה-brief. כל שינוי במחולל מאומת מול reference frozen (רגנרציה byte-identical).
  • Audit trail: כל נתון → מקור (source_file + page/row); כל prompt → חותמת sha; כל גרסת דו"ח → V-number; כל ארטיפקט נגזר מסומן DO NOT EDIT BY HAND.
  • מצב גנריות הסקריפטים (חוב 18-האזורים) מתועד בטבלה ב-ORCHESTRATION.md. סקריפטי legacy (V4-era) ב-scripts/legacy/ — ARCHIVED.

הרכיבים הניתנים לשימוש-חוזר (מחוץ לתחום זיהום מי תהום)

  • מנוע ניתוח דטרמיניסטי (מגמות/חומרה) → כל סדרת-זמן סביבתית
  • דפוס LLM מבוקר (prompt בנוי + שערים) → כל משימת ניסוח/סיווג עקבית
  • מחולל דו-קהלי (פנימי+ציבורי מאותו מקור) → כל תוצר עם גרסת-פרסום
  • שכבת QA (audit trail, אנונימיזציה, staleness) → כל זרימה רגולטורית
לסיכום הנדסי: הערך אינו במחולל-דו"ח חד-פעמי, אלא בארכיטקטורה — הפרדה נקייה בין דטרמיניזם ל-LLM, חוזים אכופים בין שלבים, ושכבת בקרה שמונעת drift, staleness ודליפה. זה מה שהופך את הכלי ממדגם-הוכחה (POC) לרכיב תשתית שניתן לתחזק, להרחיב ל-18 אזורים, ולהטמיע במערכות הרשות.
